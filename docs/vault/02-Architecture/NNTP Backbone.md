# NNTP Backbone

OME does not invent a new distributed protocol. It reuses **NNTP** — the
Network News Transfer Protocol — as both its storage layer and its
peer‑to‑peer replication layer.

The implementation is [InterNetNews (INN)](https://www.isc.org/othersoftware/#INN),
a mature open‑source Usenet server.

## Why NNTP?

- **Distributed by design.** Many peers, eventual consistency, no central
  authority. The exact shape OME needs.
- **Content‑addressable.** Every article has a unique `Message‑ID`
  header, so the same article replicated across ten peers is still one
  logical article.
- **Rich article model.** Articles can carry MIME enclosures, so we can
  attach the original source JSON alongside the OME JSON.
- **Store and forward.** Articles are durable; a peer that was offline
  for a week can catch up by exchanging `Message‑ID` lists.
- **Boring and stable.** No novel crypto or consensus layer to debug.

See [[../09-Decisions/ADR-0001 NNTP as Backbone]] for the decision
record.

## Newsgroup conventions

Each plugin declares one or more newsgroups. By convention names start
with `ome.`, followed by the source name:

| Newsgroup                     | Source                                    |
|-------------------------------|-------------------------------------------|
| `ome.eric`                    | Education Resources Information Center    |
| `ome.oercommons` / `ome.oer`  | OER Commons                               |
| `ome.openlibrary`             | Open Library (Internet Archive)           |
| `ome.qubes`                   | QUBES Hub                                 |
| `ome.whg`                     | World Historical Gazetteer                |
| `ome.pressbooks`              | Pressbooks Directory                      |
| `ome.openstax`                | OpenStax                                  |
| `ome.prelinger`               | Prelinger Archives (Internet Archive)     |
| `ome.open_education_network`  | Open Education Network                    |

Plus INN’s built‑in control groups (`control.newgroup`, `control.rmgroup`,
etc.) and `local.general` / `local.test` for development.

## Article shape

An OME article is a MIME multipart message built by
[`server/nntp_article.py`](../../../server/nntp_article.py):

```
Subject: <EducationResource.title>
From:    <librarian contact>
Newsgroups: ome.<source>
Message-ID: <generated by INN at post time>

<body = OME EducationResource JSON>

(attachment) application/json : <source>_item.json
(attachment) application/json : <source>_ome_item.json
```

The body is the OME JSON. The first attachment is the raw source JSON
(in case plugins or clients need fields OME didn’t map). The second
attachment is the OME JSON again, explicitly typed as `application/json`
so strict clients don’t have to parse the MIME body.

## Two‑node demo topology

`docker-compose.yml` stands up two INN servers side by side so
contributors can watch replication on one laptop:

- **Austin** — `inn2-docker` on host port `119`
- **Boston** — `inn2-docker` on host port `1119`

Neither server has real peering configured; `scripts/nntp_sync.py`
impersonates a peer by listing articles on Austin, diffing by
`Message‑ID`, and re‑posting what's missing on Boston every 5 seconds.
This is a temporary workaround for
[greenbender/inn-docker#26](https://github.com/greenbender/inn-docker/issues/26).
See [[../06-Operations/NNTP Sync]].

## Useful commands inside an INN container

```bash
docker exec -it ome-internetnews-server-austin-1 sh -c \
  "ctlinnd newgroup ome.eric"          # create a group manually
docker exec -it ome-internetnews-server-austin-1 sh -c \
  "ctlinnd reload active 'ome plugin add'"   # rescan active file
```

Full operator procedure in [[../06-Operations/Running INN]].
